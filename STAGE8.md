# Этап 8 — hardening и производительность

Детализация этапа 8 из [PLAN.md](PLAN.md) (§4). Оценка: **~2 недели**
(9–10 рабочих дней; необычный профиль — примерно половина времени это
стендовые прогоны и разбор их результатов, а 24-часовой long-run добавляет
календарного времени, не рабочего).

**Цель:** превратить функционально полный агент (после этапа 7 — продукт,
который разворачивается за 5 минут) в **v1.0 с доказательствами**:
подтверждённый бюджет производительности (<3% TPS, <1 CPU core на
~50k qps), задокументированная точность против `pg_stat_statements` и
лога PostgreSQL, отфаззенный парсер, проверенная матрица ядер
(5.15/6.1/6.8+), доказанное отсутствие утечек на 24-часовом чурне.
Этап почти не добавляет функциональности — он добавляет **числа, тесты
и уверенность**. Новый код: CO-RE-ветки для ядер < 6.x в `src/bpf/`,
фазз-таргеты, стендовые скрипты; пайплайн (`src/proto/`, `src/norm/`,
`src/metrics/`, `src/export/`) меняется только багфиксами по результатам
фаззинга и long-run.

**Не входит в этап:**
- **оптимизации ради оптимизаций** — любая заготовка из таблицы Р54
  берётся в работу только при провале бюджета или по данным профиля;
  дефолт этапа — «бюджет сходится, не трогаем»;
- **многопоточность как фича** (Р8 — «не раньше этапа 8») — последний
  резерв при провале бюджета, отдельное решение с новым Р-номером;
- **перф-замеры в CI** — числа снимаются на стенде с фиксированными
  условиями; CI шумный, continuous benchmarking — v1.1+ по потребности;
- **OSS-Fuzz** — интеграция имеет смысл после публикации репозитория;
  v1 ограничивается libFuzzer в своём CI (Р51);
- **unix domain sockets, arm64** — known gaps v1.1 (PLAN.md §5, STAGE7);
- **ядра 5.8–5.14** — ringbuf там есть, но матрица начинается с 5.15;
  floor фиксируется в README (Р52), «5.8+» из PLAN.md §2 уточняется.

**Долги предыдущих этапов, закрываемые здесь:**

- **этап 0** (0.1): «запуск самих BPF-программ в CI (vmtest/qemu) —
  этап 8» — закрывается (Р52, задача 8.4);
- **этап 0** (0.3): ветки CO-RE для старых ядер — «каркас заложить
  сейчас, наполнение (проверка на 5.15) — этап 8»; `docs/notes-iov.md`
  прямо назван «входными данными для матрицы ядер этапа 8» (Р52);
- **этап 1**: «оверхед „на глаз" не изменился (формальный бюджет —
  этап 8)» — формальный бюджет снимается в 8.1 (Р49);
- **этап 3** (3.6): фазз-харнесс `lk_pg_fuzz_one` заложен, «полноценный
  libFuzzer — этап 8» — закрывается (Р51);
- **этап 4**: харнесс `lk_norm_fuzz_one` — то же (Р51); «формальная
  сверка с pg_stat_statements — этап 8» (Р25/Р26) — закрывается (Р50);
- **этап 6** (Р40): «замер uprobe-оверхеда — этап 8» — TLS-конфигурация
  в бенчмарке (Р49);
- **этап 7** (Р47): requests/limits в DaemonSet — «рабочая гипотеза,
  уточняет этап 8» — уточняются числами из 8.1;
- **отложенные оптимизации** этапов 2–6 (атрибуция потерь направлению
  Р9, динамический флип capture_mode Р21, кэш fp и пиннинг горячих fp

**Отклонение от PLAN.md §3:** новых top-level каталогов нет — всё

добавляются `perf.md` и `accuracy.md`.

## Ключевые проектные решения

Нумерация продолжает Р1–Р48 из [STAGE1.md](STAGE1.md) …

### Р49. Overhead-бенчмарк: парные ABAB-прогоны, валидность = ноль потерь, бюджет как ворота v1.0

Бенчмарк — скрипт (`tests/bench/run.sh`), не ручные команды:
повторяемость важнее удобства разового запуска. Условия фиксируются
скриптом же: `cpupower frequency-set -g performance`, проверка, что на
хосте нет посторонней нагрузки, все параметры прогона — в шапке отчёта.

**Конфигурации** (одна нагрузка, пять окружений):

| # | Окружение | Что изолирует |
|---|---|---|
| A | агент не запущен | baseline |
| B | агент запущен, фильтр мимо (`--port 1`) | стоимость самих fentry-проб без пайплайна |
| C | агент, plaintext-захват | полный путь: пробы + ringbuf + пайплайн |
| D | агент, TLS-захват (pg-ssl-test, `sslmode=require`) | + uprobe-канал (долг Р40) |
| E | C + спаны 1:100 + OTLP в dummy-коллектор | + экспорт (Р31/Р32) |

**Нагрузка**: pgbench select-only `-M prepared` (потолок qps — целимся
в ~50k) и TPC-B (запись) — оба по требованию PLAN.md §4 с `-c 100+`
(`-c 128 -j $(nproc)`); scale подобрать так, чтобы база влезала в
`shared_buffers` (CPU-bound, не IO-bound — иначе меряем диск, а не
агента). Грабли стенда учтены: pgbench из docker (`--network host`),
подключение по IP контейнера — docker-proxy не должен попадать в захват.

**Протокол замера**: прогрев, затем ≥5 пар чередованием ABAB (baseline и
конфигурация перемежаются — дрейф машины размазывается по обоим), по
60–120 с; берём медианы, фиксируем разброс. **Прогон валиден только
если** `latkit_ringbuf_dropped_total` не вырос и resync = 0 — цифры
overhead при деградировавшем захвате бессмысленны (агент, теряющий
события, «дешевле», чем работающий).

**Метрики**: TPS и p50/p95 latency из pgbench (`--progress`, `-r`); CPU
агента — Δ`process_cpu_seconds_total` (self-метрика Р27) / wall,
нормированный на qps → «cores per 50k qps»; CPU postgres — из cgroup
контейнера (кросс-проверка, что агент не перекладывает стоимость на
захватываемого).

**Ворота v1.0** (PLAN.md §4/§6): ΔTPS(C vs A) и ΔTPS(D vs A) < 3%;
CPU агента < 1 core при ~50k qps. B — диагностика, не ворота: отделяет
стоимость проб (неустранимую без смены механизма) от стоимости пайплайна
(оптимизируемой по Р54). Если стенд не выдаёт 50k qps — нормировка
CPU/qps с задокументированной экстраполяцией, а не «не смогли — не
меряем».

Вместе с числами в каждом C/D-прогоне снимается `perf record -g` агента —
профили уходят в `docs/perf.md` и служат входом для go/no-go Р54, даже
если ворота пройдены. Результаты: `docs/perf.md` (числа, условия,
команды, профили), краткая сводка в README со ссылкой, уточнённые
`resources` в DaemonSet (Р47).

### Р50. Точность: три источника, join через собственный нормализатор, смещение — задокументированная величина

Сверяем на одной контролируемой нагрузке три измерения одного и того же:

1. **агент** — гистограммы (`--dump-metrics` после прогона);
2. **`pg_stat_statements`** — `calls` / `mean_exec_time` (сброс перед
   прогоном);
3. **csvlog** с `log_min_duration_statement=0` — пер-стейтментные
   `duration:` строки → из них строятся честные перцентили (у
   pg_stat_statements перцентилей нет — только mean/stddev/min/max,
   поэтому p50/p95 сравниваем именно с логом).

**Join по нашему нормализатору**: текст стейтмента из csvlog прогоняется
через `lk_norm_sql` (утилита `logjoin` на базе `src/norm/` — тот же код,
что в агенте) → fp → ряд агента. `queryid` pg_stat_statements для join
не используется — у него своя нормализация, расхождения в ней дали бы
ложные несовпадения; pgss остаётся sanity-проверкой calls/mean по
вручную сопоставленным топ-запросам.

**Протокольная тонкость** (методологически важная): основная сверка —
**simple protocol** (`-M simple`): в логе `duration` покрывает весь
стейтмент и сопоставим с моделью агента Q→Z (Р16/Р25). Extended —
отдельный прогон с оговоркой: PG логирует parse/bind/execute раздельно,
агент меряет Bind→ReadyForQuery — сравниваем с суммой фаз и фиксируем
методику в отчёте.

**Дискретизация**: перцентиль агента вычисляется из лог-сетки бакетов
(Р24) и несёт собственную ошибку ≤ шага сетки. Чтобы не смешивать ошибку
сетки с ошибкой измерения, лог-длительности прогоняются через **ту же
сетку** — сравниваются одинаково дискретизированные величины; отдельно
показывается сырой (недискретизированный) перцентиль лога.

**Ожидание и критерии**: агент ≥ лог систематически (в его тайминг
входят ядро/сеть/очередь до бэкенда — модель измерения из PLAN.md §1);
на loopback смещение — десятки–сотни мкс. Приёмка: `count` совпадает
**точно** (прогон без потерь — иначе невалиден, как в Р49); p50/p95 в
пределах 5% относительных для запросов ≥ 1 мс; для sub-ms запросов
фиксируем абсолютное смещение как характеристику, а не как «ошибку».
Дополнительно: `latkit_query_rows_total` против известных размеров
ответов, `latkit_query_errors_total{sqlstate}` против инъецированных
ошибок (`select 1/0` → 22012, нарушение unique → 23505).

Результат — `docs/accuracy.md` (методика, числа, оговорки) + раздел
README «модель измерения» (Р44) дополняется измеренным смещением: это
прямая митигация риска «точность» из PLAN.md §5.

### Р51. Fuzzing: libFuzzer на трёх таргетах, корпус в репо, инварианты вместо «не падает»

Заделы этапов 3/4 (`lk_pg_fuzz_one`, `lk_norm_fuzz_one` — вход уже
оформлен одной функцией) доводятся до полноценного libFuzzer, плюс
третий таргет:

- **fuzz_pg** — `bytes → lk_msg → pg-парсер` (задел 3.6);
- **fuzz_norm** — `lk_norm_sql` на произвольных байтах (задел этапа 4);
- **fuzz_pipe** — структурный: вход интерпретируется как сценарий
  событий `{dir, len, gap, close, …}` → conn_table + framer + decode +
  парсер целиком. Ловит межмодульное, что не видно по-таргетно:
  ресинхронизацию после дыр (Р10/Р19), startup-фазу, capture_mode-пути —
  исторически самые хрупкие швы.

**Сборка**: отдельный CMake-профиль (`-DLATKIT_FUZZ=ON`, только clang):
`-fsanitize=fuzzer,address,undefined`. **Сид-корпус**: байты сообщений
из `fixtures_gen`, живые `.lkt`-трассы, найденные ранее краевые случаи;
словарь `pg.dict` — теги сообщений v3, SQL-ключевые слова, магические
числа (`SSLRequest`, `CancelRequest`). Корпус минимизируется
(`-merge=1`) и живёт в `tests/fuzz/corpus/` — он же регрессионный набор.

**Проверяем инварианты, не только отсутствие крэша** — ASAN ловит память,
`assert` в харнессе ловит логику: парсер всегда терминирует и не читает
за границей cursor'а (Р18); «обрезано» ≠ «битое» (`trunc` не
взаимоисключён с валидным выходом); нормализатор выдаёт `text_len ≤ cap`
и стабильный fp (два прогона — один результат); framer никогда не
эмитит сообщение через дыру (инвариант Р19).

**CI**: на PR — corpus-регрессия (прогон корпуса без фаззинга, секунды)
плюс 60 с фаззинга на таргет; nightly по schedule — 15 мин на таргет,
крэш-репро уходит артефактом. Каждая находка после фикса → регрессионный
вход в корпус (и, где осмысленно, `.lkt`-фикстура).

### Р52. Матрица ядер: vmtest + готовые BPF-CI-ядра, smoke без docker, floor = 5.15 с BTF

**Матрица**: 5.15 / 6.1 / 6.8 / актуальное stable — через
[vmtest](https://github.com/danobi/vmtest) (qemu-обёртка, шарит rootfs
хоста) с готовыми ядрами из libbpf CI-сборок — не собираем ядра сами.
GitHub Actions Linux-раннеры дают `/dev/kvm`; дополнительно тот же
скрипт гоняется локально (`tests/kernel/smoke.sh` не знает, VM это или
железо). Джоба `kernels.yml` — на push в master, nightly и по лейблу на
PR: гонять полную матрицу на каждый PR дорого и не нужно.

**Что крутится в VM** — никакого docker/postgres: статический бинарник
(Р45) + два крошечных хелпера:

- `pgstream` — TCP-пара, проигрывающая байтовые потоки из
  `fixtures_gen` через реальный сокет 127.0.0.1:5432 (loopback в VM
  чист — docker-proxy-граблей нет по построению);
- `tlspipe` — то же через libssl хостового rootfs (vmtest шарит fs):
  реальный SSL-handshake + фикстурные байты в `SSL_write` — uprobe-канал
  (этап 6) подтверждается на каждом ядре матрицы, включая
  `SSL_set_fd`-walk моста Р37 (риск из STAGE6: «CO-RE-обход fd→sock
  хрупок между версиями — проверка на матрице ядер»).

Ассерты smoke: OPEN/CLOSE видны, распознано ровно N запросов, ноль
parse errors, ноль resync и — главный детектор CO-RE-регрессий —
**ноль `iter_unsupported`**.

**CO-RE-ветки < 6.x** — наполнение каркаса из этапов 0/1
(`iter_first_seg`; `docs/notes-iov.md` — входные данные): на 5.15 нет
`ITER_UBUF` (появился ~6.0) — основной путь IOVEC; `iov` vs `__iov`
(переименование ~6.4) — через `bpf_core_field_exists`; наличие UBUF —
через `bpf_core_enum_value_exists`. Правка только в `src/bpf/` — формат
событий не меняется.

**Floor фиксируется**: v1 поддерживает **5.15+ с BTF** — то, что реально
проверяется матрицей. PLAN.md §2 говорил «5.8+ (ringbuf), целимся в
5.15+» — README после этапа формулирует однозначно: 5.15+, ниже — не
тестируется и не обещается. **Graceful degradation без BTF** = внятный
отказ, не тихие нули: одно ядро 6.1 без `CONFIG_DEBUG_INFO_BTF`
собирается однократно (кэшируется в CI) → negative-тест: агент завершает
работу с «kernel 5.15+ with BTF required», без сегфолта и без запуска
«вслепую».

### Р53. Утечки: valgrind поверх ASAN в nightly, long-run 24h с чурном и индуцированными потерями

**Санитайзеры** ASAN/UBSAN на unit/replay уже в CI (этапы 2–4) —
остаются как есть. Добавляется **valgrind memcheck на replay-тестах** в
nightly (на PR — медленно): его uninit-value-анализ покрывает
MSAN-территорию без пересборки всех зависимостей; suppressions-файл —
рядом в `tests/`, каждая запись с комментарием почему.

**Long-run 24h** (`tests/longrun/run.sh`, на dev-стенде):

- умеренный постоянный pgbench + **чурн-цикл**: короткоживущие
  соединения (psql-петля с реконнектом раз в секунды) — LRU/idle-sweep
  таблицы соединений (Р12) под непрерывным давлением;
- параллельно TLS-контейнер (pg-ssl-test) — attach/detach uprobes и
  мост Р37 в долгом цикле;
- **индуцированные потери** раз в час: `pkill -STOP -x latkit; sleep;
  pkill -CONT` (рабочий способ со стенда — естественным путём потери не
  спровоцировать) — путь «дыра → resync → восстановление» проверяется
  десятками итераций, а не однократно;
- периодический рестарт postgres-контейнера — synthetic OPEN, ре-скан
  libssl (Р39), ре-резолв cgroup (Р48) в одном флаконе;
- «свидетель» — prometheus стенда, скрейпящий self-метрики агента: RSS,
  `latkit_conns_active` против реальности, `latkit_metric_series` ≤
  потолка K (Р23), fd-count процесса (скриптом из `/proc`), dropped/
  resync растут **только** в индуцированные окна.

**Критерии**: агент жив все 24 ч; RSS выходит на плато и стабилен
(< +5% за последние 12 ч); fd не растут; после каждого индуцированного
окна потерь захват возвращается к нулевым resync; карты ядра не растут
(`bpftool map` до/после). Краш или монотонный рост → фикс → повтор
(календарный запас на один повтор заложен в оценку этапа).

### Р54. Оптимизации — только по данным; заготовки получают go/no-go, а не вечное «потом»

Этапы 2–6 накопили заготовленные оптимизации. Правило этапа: заготовка
берётся в работу **только если** провалены ворота Р49 или профиль
показывает заметную концентрацию (ориентир: ≥10% CPU агента в одном
месте). Каждая строка таблицы получает в 8.6 явное решение по данным
8.1–8.5 — «go» (делаем в 8.6), «v1.1» (есть сигнал, не блокирует v1.0)
или «drop» (данные показали ненужность):

| Заготовка | Откуда | Триггер |
|---|---|---|
| Второй поток / lock-free агрегация | Р8, PLAN §4 этап 2 | пайплайн-поток — узкое место бюджета |
| Атрибуция потери seq направлению (грязнить одно, не оба) | Р9 | счётчик resync показывает заметный уровень на чистой нагрузке |
| Динамический флип capture_mode FULL↔HEADERS | этап 3 (Р21, отложено «с измерениями») | стоимость копирования тел доминирует в B→C дельте |
| Кэш fp по указателю/длине сырого текста | этап 4 (риски) | нормализация ≥ 10% CPU в профиле |
| Пиннинг горячих fp против counter reset | этап 4 (риски) | жалобы на reset в практике — сигнала пока нет |
| Шардирование/RCU реестра метрик | Р26 | конкуренция дампа и агрегации в профиле |
| Инкрементальный дамп метрик кусками (`lk_metrics_iter`) | этап 5 (риски) | дамп реестра блокирует пайплайн → потери на всплеске при скрейпе |
| Только-`SSL_read` + реконструкция таймингов | Р40 | uprobe-оверхед (D vs C) — доминирующая часть дельты |

Многопоточность — отдельно: это не «оптимизация по ходу», а
архитектурное решение; если до неё дойдёт — новый Р-номер, своя задача,
пересмотр Р8/Р26. Ожидание (по замерам этапов 4–5): не дойдёт.

## Риски этапа

| Риск | Митигация |
|---|---|
| Шум стенда сопоставим с искомой дельтой 3% | ABAB-чередование, медианы ≥5 пар, фиксированный governor, чистый хост; если разброс > эффекта — удлинить прогоны и зафиксировать доверительный интервал, а не «на глаз» |
| Стенд не выдаёт 50k qps — бюджет CPU не проверить буквально | нормировка cores/qps на достижимом максимуме + задокументированная экстраполяция; select-only `-M prepared` как потолочный режим |
| Бюджет провален | заготовки Р54 ранжированы по дешевизне, профиль уже снят в 8.1; худший случай (многопоточность) — осознанное отдельное решение, календарно за пределами оценки этапа |
| Сверка с pg_stat_statements разъедется из-за чужой нормализации | join только через собственный нормализатор по csvlog; pgss — sanity, не эталон; протокольная тонкость extended учтена методикой |
| Фазз найдёт глубокое — переписывание куска парсера | bounded cursor (Р18) локализует фиксы; инварианты в харнессах сужают триаж; время на находки заложено в 8.3; корпус не даёт регрессий вернуться |
| vmtest/KVM в GH Actions капризен | матрица не на PR-пути (push/nightly/лейбл) — не блокирует разработку; тот же `smoke.sh` гоняется локально без CI |
| CO-RE-ветки для 5.15 написаны «по документации» и не работают | проверка — само ядро 5.15 в матрице, ассерт `iter_unsupported=0`; `notes-iov.md` ведётся с этапа 0 именно для этого |
| Long-run упадёт на 20-м часу — сутки потеряны | метрики пишутся в prometheus непрерывно — разбор возможен без повтора начала; календарный запас на один повтор; чурн/потери агрессивнее прода — падение здесь дешевле падения у пользователя |
| valgrind шумит ложными срабатываниями | suppressions с обоснованием каждой записи; только nightly — не блокирует PR; ASAN остаётся быстрым первым эшелоном |
| Индуцированные потери (`SIGSTOP`) — не то же, что потери под реальной перегрузкой | это проверка пути восстановления, не модели потерь; реальная перегрузка дополнительно покрывается пиковыми конфигурациями бенчмарка с контролем dropped |
