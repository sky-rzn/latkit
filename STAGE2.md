# Этап 2 — userspace-пайплайн

Детализация этапа 2 из [PLAN.md](PLAN.md) (§4). Оценка: **~1 неделя**
(5–7 рабочих дней; буфер на крайние случаи reassembly заложен в задачах).

**Цель:** превратить поток сырых ringbuf-событий этапа 1 в поток **целых
сообщений протокола PG** (`тип + len + body`) на соединение, с честной
пометкой «грязных» участков после потерь. Плюс инфраструктура, на которой
это живёт: event loop на epoll, таблица соединений в userspace. На выходе —
callback-API, в который этап 3 (парсер) подключается без перекройки пайплайна.

**Не входит в этап:** семантика сообщений — state machine запросов,
сопоставление ответов, latency (этап 3); нормализация SQL и метрики (этап 4);
HTTP `/metrics` (этап 5 — но event loop проектируем так, чтобы серверный fd
воткнулся без переделки); TLS-канал через uprobes (этап 6 — здесь только
детект TLS-соединений и отбрасывание их шифрованного socket-трафика);
многопоточность и lock-free (профилировать в этапе 8, пока — один поток).

**Долги этапа 1, закрываемые здесь** (помечены в коде):

- `seqtrack` — «заготовка „грязного“ флага, не полноценная conn-table»
  (`seqtrack.h`): поглощается таблицей соединений, включая уборку записей
  соединений с потерянным `CONN_CLOSE`;
- инвариант честного `total_len` (`latkit.h`, Р6 этапа 1) наконец
  используется по назначению: reassembler продвигает поток на `total_len`
  и знает точный размер каждой «дыры»;
- цикл `ring_buffer__poll(100 ms)` + ручной таймер статистики в `main.c` —
  заменяется на epoll-петлю.

**Не** закрывается здесь: гонка read-modify-write в `set_cap_headers`
(`main.c`) — пересматривается вместе с реальной политикой capture_mode в
этапе 3.

## Ключевые проектные решения

Нумерация продолжает Р1–Р7 из [STAGE1.md](STAGE1.md).

### Р8. Один поток, свой epoll; логика — чистые функции

Пайплайн остаётся однопоточным: агент создаёт собственный
`epoll_create1()` и регистрирует в нём:

- `ring_buffer__epoll_fd(rb)` — готовность ringbuf (по срабатыванию —
  `ring_buffer__consume(rb)`, без таймаута);
- `timerfd` — периодика: печать статистики (10 с) и sweep таблицы
  соединений (60 с);
- `signalfd(SIGINT|SIGTERM)` — корректный shutdown вместо
  `volatile sig_atomic_t` (сигналы блокируются, доставка — через fd).

Этап 5 добавит в тот же epoll listen-fd HTTP-сервера — поэтому петля сразу
пишется как маленький диспетчер `fd → handler`, а не три захардкоженных `if`.

Вся содержательная логика (conn table, framing) — чистые функции без I/O,
как `decode.c`/`seqtrack.c` в этапе 1: unit-тесты кормят их синтетическими
событиями, агент — реальными. Локи не нужны вовсе; вопрос «агрегация
lock-free или под мьютексом» из PLAN.md откладывается до появления
второго потока (не раньше этапа 8).

### Р9. Streaming-framer: поток = последовательность «байты | дыра»

Ключевая идея этапа. Reassembler **не** хранит буфер потока на направление
и не «склеивает фрагменты» в классическом смысле. Каждое направление
соединения — это поток из двух примитивов, которые события этапа 1
порождают детерминированно:

- `bytes(p, n)` — захваченные байты (`payload[0..cap_len)` чанка);
- `hole(n)` — дыра **известного размера**: срез бюджетом захвата
  (`LK_F_TRUNC`: `total_len − Σ cap_len` вызова, недостающее между чанками
  по `off`).

Это возможно потому, что этап 1 гарантирует: `total_len` всегда честный,
бюджеты режут только `cap_len`. Порядок примитивов внутри направления
гарантирован: `tcp_sendmsg`/`tcp_recvmsg` на одном сокете сериализованы
socket lock'ом, чанки одного вызова submit'ятся последовательно, ringbuf
сохраняет порядок submission. Значит, per-direction поток самоописан —
буферизация нужна только для незавершённого *сообщения*, не для потока.

Дыры делятся на два принципиально разных класса:

1. **Известные** (усечение бюджетом): позиция потока точна. Если заголовок
   сообщения (5 байт) захвачен, тело любой длины пропускается арифметикой
   по `len` — усечение **не десинхронизирует** фрейминг. Дыра, накрывшая
   заголовок, — синхронизацию теряет (позицию следующего сообщения внутри
   дыры узнать не из чего) → направление «грязное».
2. **Неизвестные** (потерянные события ringbuf): дыра в `seq`. Сколько байт
   потока пропало — неизвестно в принципе (мог потеряться целый вызов) →
   направление «грязное», нужна ресинхронизация (Р10).

Дыра в `seq` — консервативно грязнит **оба** направления: `seq` общий на
соединение, атрибуция потери направлению по косвенным признакам — возможное
уточнение позже (этап 8), не сейчас. Ложные срабатывания от кросс-CPU
реордеринга соседних seq (задокументированы в `seqtrack.h`) дают лишний
ресинк — терпимо, считается счётчиком, пересмотрим, если счётчик покажет
заметный уровень.

### Р10. Framing знает рамки PG v3, но не семантику

Чтобы выделять сообщения и ресинхронизироваться, framer'у нужен минимум
знаний о протоколе — фиксируем ровно его, всё остальное — этап 3:

- **normal-фрейминг**: `тип(1) + len(4, BE, включает себя, не включает
  тип) + body(len−4)`; `len < 4` или `len > 2^30` — повреждение → dirty;
- **startup-фрейминг** (первое сообщение frontend, типа нет):
  `len(4) + code(4) + …`; по `code` различаются `StartupMessage`
  (0x00030000), `SSLRequest` (80877103), `GSSENCRequest` (80877104),
  `CancelRequest` (80877102, соединение сразу закрывается);
- **однобайтовый ответ** backend на SSL/GSSENC-request: `'S'`/`'N'`/`'G'` —
  единственное нетипизированное сообщение backend. Флаг «ждём однобайтовый
  ответ» ставится на соединении при виде SSLRequest: порядок ringbuf
  глобальный, recv(SSLRequest) в ядре happens-before send-ответа, так что
  кросс-направленный флаг корректен;
- ответ `'S'` → соединение помечается TLS: framing выключается, дальнейшие
  socket-события по нему молча отбрасываются (источником станет
  uprobe-канал этапа 6). Опционально — сразу переводить такое соединение в
  `LK_CAP_HEADERS` через карту `conns`, чтобы не гонять шифрованные байты
  через ringbuf;
- **якорь ресинхронизации**: backend — паттерн
  `'Z' 00 00 00 05 [I|T|E]` (ReadyForQuery) в захваченных байтах, framing
  возобновляется со следующего за ним байта; frontend — граница вызова
  (`off == 0`), где первый байт ∈ допустимых frontend-типов
  (`Q P B E S X C D F H d c f p`) и `len` правдоподобен. Семантику 'Z' мы
  при этом не трогаем — только используем как маркер границы.

Синтетические соединения (`LK_F_SYNTHETIC` — startup не видели) начинают
жизнь сразу в состоянии dirty и входят в поток через тот же ресинк — один
механизм на оба случая.

Наружу framer отдаёт сообщения через callback (см. «API»), с флагами
`BODY_TRUNC` (собран только префикс тела) и `AFTER_RESYNC` (первое сообщение
после ресинка — парсер этапа 3 знает, что контекст до него потерян).

### Р11. Память ограничена по построению

Следствие Р9: буферизуется только незавершённое сообщение, и то не целиком:

- заголовок — до 5 байт (нормальный) / до 8 (startup);
- тело — префикс до `LK_MSG_BODY_MAX = 16 КиБ` (больше, чем нужно этапу 4
  для фингерпринта по префиксу), остаток тела проматывается по `len`
  (состояние SKIP), включая известные дыры.

Итого worst case на направление ≈ 16 КиБ, аллоцируется лениво при первом
частичном сообщении и освобождается на границе сообщений. Никаких буферов
«на весь поток», никакого роста на больших `DataRow`-потоках. При 65536
соединений (потолок таблицы) worst case ≈ 2 ГиБ, но реальный steady state —
около нуля: частичное сообщение живёт от чанка до чанка.

### Р12. Таблица соединений: поглощает seqtrack, LRU + idle sweep

`conn_table.c`: hash-таблица cookie → `struct lk_conn` (chained, по образцу
`seqtrack`, который она поглощает — `last_seq` переезжает в состояние
соединения, `seqtrack.c` удаляется, его тесты адаптируются):

- создание — по `CONN_OPEN` (обычному или synthetic) либо лениво по первому
  data-событию с неизвестным cookie;
- удаление — по `CONN_CLOSE` (частичные сообщения отбрасываются, буферы
  освобождаются);
- **страховка от утечек** (долг этапа 1): idle-таймаут — sweep раз в 60 с
  выселяет соединения без событий дольше `--conn-idle-timeout` (дефолт
  600 с); плюс жёсткий потолок `--max-conns` (дефолт 65536, в тон карте
  ядра) с LRU-вытеснением по last_activity;
- если выселили живое соединение и трафик возобновился — запись создаётся
  заново как dirty (ядро synthetic OPEN не пришлёт: в карте `conns` ядра
  запись жива и `LK_CS_OPEN_SENT` стоит) → ресинк по Р10. Данные не
  теряются, метрики честны.

### Р13. Таймстемп сообщения — событие первого байта заголовка

Сообщению приписывается `ts_ns` события (чанка), принёсшего первый байт его
заголовка. Несколько сообщений в одном вызове send/recv делят один
таймстемп — это принципиальный предел метода: **точность латентности =
границы syscall'ов**, что честно фиксируем в docs (модель latency PLAN.md
меряет от recv-вызова с `Q` до send-вызова с `Z` — для неё этого достаточно).
Часы — `bpf_ktime_get_ns` (CLOCK_MONOTONIC), в стенные часы конвертирует
этап 4/5 при экспорте.

### Р14. Трассы: record/replay сырых событий

Отладка reassembly по live-трафику мучительна, а фикстуры этапу 3 нужны уже
сейчас. Добавляем:

- `--record FILE` у агента: каждая ringbuf-запись пишется в файл как есть,
  с length-префиксом (`magic "LKT1"`, далее `{u32 size, байты записи}`);
- replay-харнесс в тестах: читает файл и кормит записи в тот же
  `handle_event`-путь (decode → conn table → framer), что и live-агент.

Один и тот же код обрабатывает живой и записанный поток; фикстуры для
unit-тестов этапа 3 (simple, extended, pipeline, ошибки, COPY) снимаются
одной командой с psql/pgbench. Это дешевле и точнее pcap: события уже несут
conn_id, направления, таймстемпы и дыры.

## API между модулями (эскиз)

Контракт framer → потребитель (этап 2 — логгер `--messages`; этап 3 —
`pgproto.c`):

```c
/* lk_msg.flags */
#define LK_MSG_BODY_TRUNC   (1 << 0) /* body_cap < len-4: собран только префикс */
#define LK_MSG_AFTER_RESYNC (1 << 1) /* первое сообщение после потери синхры */
#define LK_MSG_STARTUP      (1 << 2) /* startup-фрейминг: type=0, body с code */

struct lk_msg {
    __u64 ts_ns;    /* событие первого байта заголовка (Р13) */
    char  type;     /* 'Q','Z',...; 0 для startup-сообщений */
    __u16 flags;    /* LK_MSG_* */
    __u32 len;      /* заявленная длина (поле протокола, без байта типа) */
    __u32 body_cap; /* байт body реально доступно (префикс) */
    const __u8 *body;
};

struct lk_msg_sink {
    void *ctx;
    void (*on_msg)(void *ctx, struct lk_conn *c, enum lk_dir dir,
                   const struct lk_msg *m);
    void (*on_conn_open)(void *ctx, struct lk_conn *c);  /* + tuple, synthetic */
    void (*on_conn_close)(void *ctx, struct lk_conn *c); /* уборка этапа 3 */
    void (*on_resync)(void *ctx, struct lk_conn *c, enum lk_dir dir);
};
```

Состояние framer'а на направление (живёт в `struct lk_conn`):

```c
enum lk_frame_state { LK_FR_HEADER, LK_FR_BODY, LK_FR_SKIP, LK_FR_DIRTY };

struct lk_frame {
    enum lk_frame_state st;
    __u32 need;      /* байт до конца заголовка / до BODY_MAX */
    __u64 skip_left; /* SKIP: остаток тела, проматываемый по len */
    __u32 call_pos, call_total; /* позиция в текущем send/recv-вызове:
                                   off-арифметика чанков → hole() */
    __u8 *buf;       /* ленивый, ≤ 5+LK_MSG_BODY_MAX (Р11) */
    __u32 buf_len;
};
```

Вход framer'а — ровно два примитива (Р9): `lk_frame_bytes(f, p, n, ts)` и
`lk_frame_hole(f, n)`; их генерирует тонкий слой поверх `lk_ev_data`
(`off`/`cap_len`/`total_len`). Всё чистое, тестируется без BPF.

## Задачи

Разбивка соответствует будущим коммитам (`Stage 2.x: ...`).

### 2.1 Event loop на epoll, каркас модулей (~1 день)

- [x] `loop.c`: epoll-петля с таблицей `fd → handler`; timerfd (тик 1 с,
      внутри — счётчики до 10 с/60 с задач), signalfd для
      SIGINT/SIGTERM.
- [x] `events.c`: вынести из `main.c` `handle_event`, печать статистики,
      обработку записей; `ring_buffer__epoll_fd` в петлю,
      `ring_buffer__consume` по готовности.
- [x] `main.c` худеет до: parse_args → open/load/attach (как в этапе 1) →
      сборка петли → `loop_run()`.
- [x] Поведение снаружи не меняется (тот же вывод, та же статистика) —
      это чисто механический рефакторинг, отдельный коммит до содержательных
      изменений.

**Готово, когда:** вывод агента на psql-сессии байт-в-байт совпадает с
этапом 1; выход по Ctrl-C корректен (линки детачатся); `strace` показывает
epoll_wait вместо частых poll(100 ms) на простое.

### 2.2 Таблица соединений (~1–1,5 дня)

- [x] `conn_table.c` по Р12: `struct lk_conn` {cookie, tuple, flags
      (synthetic/tls/dirty), last_seq, dropped, last_activity_ns, LRU-линки,
      2×`struct lk_frame`}; операции lookup/create/remove/sweep.
- [x] Поглотить `seqtrack`: детектор дыр в seq — метод таблицы; на дыру —
      пометить оба направления dirty (Р9) и залогировать. `seqtrack.c/.h`
      удалить, `test_seqtrack.c` переписать на API таблицы (сценарии те же).
- [x] Idle sweep (60 с, `--conn-idle-timeout`, дефолт 600 с) и потолок
      `--max-conns` (дефолт 65536) с LRU-вытеснением; счётчики
      created/closed/evicted/timeout/active в userspace-статистику (та же
      10-секундная строка stderr).
- [x] Пере-создание после вытеснения — как dirty-соединение (без synthetic
      OPEN от ядра), покрыть unit-тестом.

**Готово, когда:** `pgbench -C -c 20 -T 60` (чурн) — active возвращается к
нулю, created == closed (± служебные); тест с задавленным CLOSE (replay
без CLOSE-записей) — записи уходят по таймауту; unit-тесты таблицы зелёные.

### 2.3 Streaming-framer: normal-фрейминг и дыры (~2 дня)

- [x] Слой `lk_ev_data` → примитивы: off-арифметика чанков вызова
      (`call_pos`/`call_total`), генерация `hole()` на усечение
      (`LK_F_TRUNC`) и недостающие интервалы `off`; аномалии (off назад,
      off за total) → dirty + счётчик.
- [x] `lk_frame_bytes`/`lk_frame_hole`: state machine
      HEADER→BODY→SKIP (Р9/Р11); `LK_MSG_BODY_MAX = 16 КиБ`, хвост тела —
      SKIP по `len`; дыра в HEADER → dirty; санити `len` (< 4, > 2^30) →
      dirty; эмиссия `lk_msg` с `BODY_TRUNC`, ts по Р13.
- [x] Логгер-потребитель `--messages`: строка на сообщение
      (`ts dir conn type len body_cap flags`), `--hexdump` дампит префикс
      тела. Пер-событийный вывод этапа 1 — за флагом `--events`.
- [x] Unit-тесты на синтетических событиях (builder-хелперы): сообщение в
      несколько чанков/вызовов; несколько сообщений в одном чанке; тело
      больше BODY_MAX; тело, целиком накрытое известной дырой (заголовок
      виден — синхра держится); дыра, накрывшая заголовок → dirty; чанк
      потерян внутри вызова.

**Готово, когда:** `psql -c "select 'marker', 42"` даёт лог
`startup → Q → T, D, C, Z` с корректными len; `select repeat('x',100000)`
собирается в один DataRow c `BODY_TRUNC` и правильным `len` без потери
синхры; юниты зелёные.

### 2.4 Startup-фаза, TLS-детект, ресинхронизация (~1–1,5 дня)

- [x] Startup-фрейминг frontend (Р10): StartupMessage/SSLRequest/
      GSSENCRequest/CancelRequest по `code`; после StartupMessage —
      normal-фрейминг; CancelRequest — эмиссия и ожидание CLOSE.
- [x] Однобайтовый ответ на SSLRequest: флаг «ждём байт» на соединении;
      `'S'` → пометить TLS, выключить framing, события отбрасывать
      (+ счётчик tls_conns; опционально — capture_mode=HEADERS в карту
      `conns`); `'N'` → продолжаем plaintext.
- [x] Ресинхронизация (Р10): backend — скан на `'Z' 00 00 00 05 [ITE]`
      (аккуратно с паттерном, порванным по границе событий — скользящее
      окно 6 байт); frontend — граница вызова + валидный тип + правдоподобный
      len. Первое сообщение после ресинка — `AFTER_RESYNC`; счётчик
      resync_total (застолбить имя `latkit_resync_total` для этапа 4).
- [x] Синтетические соединения стартуют dirty (один механизм с ресинком);

## Риски этапа

| Риск | Митигция |
|---|---|
| Ресинк-эвристика ловит ложный 'Z'-паттерн в данных (сдвинутый фрейминг) | скан только в состоянии dirty; санити len следующего сообщения — при мусоре сразу обратно в dirty; frontend-якорь требует границу вызова; счётчик повторных ресинков на соединение |
| Ложные dirty от кросс-CPU реордеринга seq (известный артефакт этапа 1) | терпимо: лишний ресинк, не потеря корректности; счётчик spurious-кандидатов; если уровень заметен — окно допуска ±1 seq перед пометкой |
| off-арифметика чанков разойдётся с реальностью на каком-то пути ядра | аномалии off → dirty + отдельный счётчик (не тихая порча); replay-фикстуры с реального трафика в CI |
| Idle-таймаут выселяет живые пулы соединений → волны dirty/resync | вытеснение теряет только пустой framer-стейт на границе сообщений; дефолт 600 с консервативен; счётчик timeout-вытеснений подскажет тюнинг |
| BODY_MAX 16 КиБ окажется мал для парсера этапа 3 (гигантские SQL) | по построению это уже больше бюджета захвата на вызов (8 КиБ дефолт); фингерпринт по префиксу — принятое решение PLAN.md (этап 3, «крайние случаи») |
| epoll-рефакторинг ломает что-то тонкое в жизненном цикле skeleton | задача 2.1 — отдельный механический коммит с критерием «вывод байт-в-байт», до любых содержательных правок |
