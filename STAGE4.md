# Этап 4 — нормализация запросов и метрики

Детализация этапа 4 из [PLAN.md](PLAN.md) (§4). Оценка: **~1–1,5 недели**
(6–7 рабочих дней; нормализатор и контроль кардинальности — основной объём).

**Цель:** превратить поток наблюдений `lk_query_obs` (выход парсера этапа 3)
в **ограниченный по кардинальности набор рядов метрик** — гистограммы
latency, счётчики строк/ошибок/соединений — лежащий в памяти агента в виде,
который этап 5 сериализует в Prometheus text format и OTLP без пересчёта.
Это веха **M2** (PLAN.md §6): pgbench-нагрузка → корректные
latency-гистограммы по нормализованным запросам (plaintext).

**Не входит в этап:** HTTP `/metrics` и `/healthz` (этап 5 — здесь только
текстовый дамп в файл/stderr для валидации); OTLP, спаны и exemplars с
полным текстом SQL (этап 5); TLS (этап 6); формальное сравнение точности с
`pg_stat_statements` (этап 8 — здесь только sanity-прикидка);
полноценный fuzzing нормализатора (этап 8 — харнесс закладывается здесь,
как для парсера в 3.6).

**Долги этапа 3, закрываемые здесь** (застолблены в STAGE3.md):

- выбор длительности из четырёх таймстемпов наблюдения (`ts_start`,
  `ts_first_row`, `ts_complete`, `ts_ready`) — парсер сознательно не решал
  (Р16), решает Р25;
- счётчики парсера/framer'а/ядра, живущие в 10-секундной строке статистики,
  становятся настоящими метриками с застолблёнными именами
  (`latkit_parse_errors_total`, `latkit_resync_total`,
  `latkit_ringbuf_dropped_total`, …) — Р27;
- «запрос, оборванный дисконнектом» (слепое пятно Р19) — получает счётчик;
- логгер `--queries` перестаёт быть единственным потребителем
  `lk_query_sink` — штатным становится агрегатор, `--queries` остаётся
  отладочным флагом (tee).

**Отклонение от PLAN.md §3:** вместо `src/agent/normalize.c` и
`src/agent/metrics.c` — каталоги `src/norm/` и `src/metrics/`:
`src/agent/` после этапа 3 — это «BPF, события, framing — без семантики»,
а нормализатор SQL пригодится любому SQL-протоколу (MySQL — кандидат
после v1), метрики же вообще протокол-независимы. PLAN.md поправить в
рамках задачи 4.5.

## Ключевые проектные решения

Нумерация продолжает Р1–Р21 из [STAGE1.md](STAGE1.md) /
[STAGE2.md](STAGE2.md) / [STAGE3.md](STAGE3.md).

### Р22. Нормализатор — однопроходный лексер, не парсер

Как pg_stat_statements по духу, но без парсера PG (PLAN.md §4): один
проход по байтам, конечный автомат лексера, нулевые аллокации (выход — в
буфер вызывающего). Правила, в порядке приоритета:

- **комментарии** `-- ...\n` и `/* ... */` (вложенные — PG их вкладывает!)
  → выбрасываются;
- **строковые литералы** `'...'` (escape `''`), `E'...'` (escape `\`),
  `B'...'`, `X'...'`, dollar-quoting `$tag$...$tag$` → токен `?`;
- **числа** — целые, десятичные, экспонента, `0x`/`0o`/`0b`,
  разделители `1_000_000` (PG 16+) → `?`; знак перед числом — оператор,
  не часть литерала (`-1` → `- ?`; иначе `a-1` и `a - 1` разойдутся);
- **параметры** `$1`, `$2`, … → `?` — **осознанное отклонение от
  pg_stat_statements** (он сохраняет `$N`): так
  `where id = 42` (simple), `where id = $1` (prepared) и `where id = ?`
  (JDBC-style) сливаются в один fingerprint — для top-K это желаемое
  поведение; задокументировать;
- **ключевые слова и нецитированные идентификаторы** → нижний регистр
  (PG сам сворачивает unquoted-идентификаторы к lower — семантика не
  меняется); `null`/`true`/`false` **не** заменяются на `?` (лексер не
  отличит `is null` от литерала — оставляем как есть, отклонение
  задокументировать);
- **цитированные идентификаторы** `"..."` (escape `""`) — как есть,
  регистр значим;
- **whitespace** → токен-разделитель; хвостовая `;` отбрасывается
  (`select 1` и `select 1;` — один fingerprint);
- **схлопывание списков**: на выходе токенизации последовательность
  `( ? , ? , … )` → `( ? )` — покрывает `IN (1,2,3)`; затем
  последовательность групп `( ? ) , ( ? ) , …` → `( ? )` — покрывает
  многострочный `VALUES`.

Канонический текст — токены через один пробел. **Fingerprint —
XXH3-64 по потоку токенов** (с `\0`-разделителями), считается стримингово
**независимо от текстового буфера**: если нормализованный текст не влез в
`LK_NORM_TEXT_MAX` (1 КиБ), текст усечён, но хэш продолжает считаться по
токенам до конца входа — усечение лейбла не меняет идентичность запроса.
xxHash вендорится в `third_party/xxhash/` (один header, BSD).

Вход усечённый (`LK_QO_TEXT_TRUNC`, бюджет захвата): лексер, упёршийся в
конец буфера посреди токена, завершает чисто — fingerprint по префиксу,
выходной флаг `trunc` (принятое решение PLAN.md: «фингерпринт по префиксу
с пометкой»). Разные запросы с общим префиксом длиннее бюджета захвата
(8 КиБ) сольются — принятая цена, задокументировать. `MULTI_STMT`-тексты
не режутся на стейтменты — нормализуются целиком, `;` внутри остаётся
токеном.

Вход недоверенный — те же требования, что Р18: никакой арифметики мимо
границ, лексер не может выйти за `[sql, sql+len)` по построению;
fuzz-харнесс `lk_norm_fuzz_one()` — с первого дня (полный libFuzzer —
этап 8).

### Р23. Кардинальность: реестр серий, top-K LRU, bucket `other`

Критично для Prometheus (PLAN.md §5 «взрыв кардинальности»). Три уровня
защиты:

- **словарь запросов**: hash map fingerprint → `{нормализованный текст
  (усечённый до лейбла), счётчик использования}`, ёмкость
  `K = 500` (конфигурируемо, `--top-queries`). Вытеснение — LRU;
  **вытесняемые ряды сливаются в `other`**: их суммы/бакеты прибавляются
  к рядам `query="other"` — глобальные суммы остаются монотонными, сам
  ряд `other` не убывает. Вернувшийся после вытеснения fingerprint
  начинает с нуля — для Prometheus это обычный counter reset, `rate()`
  переживает; задокументировать;
- **doorkeeper против чурна**: поток одноразовых ad-hoc запросов (миграции,
  psql-сессии людей) не должен вымывать рабочий top-K. Новый fingerprint
  при полном словаре идёт в `other` и записывается в кольцевой набор
  кандидатов (fingerprint'ы, N=2K); допуск в словарь — со **второго**
  появления. Дёшево (одна hash-проба), устраняет главный паттерн чурна;
- **вторичные измерения**: пары `(db, user)` тоже неограниченны
  (`latkit_query_duration_seconds{query, db, user, code}` — произведение!).
  Отдельный лимит `LK_MAX_SESSIONS_DIM = 32` различных пар (конфигурируемо);
  сверх лимита — `db="other", user="other"`. SQLSTATE в
  `latkit_query_errors_total{sqlstate}` — лимит 64 различных кодов, сверх —
  `other`.

**Уточнение PLAN.md:** лейбл `code` в duration-гистограмме — это
`ok | error` (2 значения), **не** сырой SQLSTATE: SQLSTATE в произведении
с `query × db × user` взорвал бы число рядов; детализация по SQLSTATE
живёт в отдельном счётчике ошибок без лейбла `query`.

Худший случай памяти реестра — явный потолок: K=500 запросов × 32 пары
(db,user) × 2 code × (~80 бакетов × 8 Б + текст ≤ 1 КиБ словаря) —
порядка десятков МиБ при полном заполнении всех комбинаций; реально
запросы размазаны по парам, а не декартово произведение. Потолок и формулу
— в README; счётчик фактического числа рядов — в self-метрики.

### Р24. Гистограмма: экспоненциальная сетка, совместимая с native histograms

Внутреннее представление одно, экспорта два (PLAN.md: «нативные
гистограммы + классические бакеты, лог-шкала 0.1ms…60s»):

- **сетка**: границы `2^(k/4)` секунд (schema=2 в терминах Prometheus
  native histograms / OTLP exponential histograms — фактор ≈1.19, ошибка
  бакетизации ≤ ±9%). Диапазон 0.1 мс … 60 с → индексы k ∈ [-53, 24),
  **77 бакетов** + underflow + overflow + `sum` + `count` — плоский массив
  ~80 × u64, ~650 байт на ряд, инкремент — O(1) по битовым операциям над
  IEEE-754 экспонентой (без `log()` на горячем пути);
- **классические бакеты** (text format этапа 5, dump этапа 4): каждая
  четвёртая граница сетки → лог-шкала с фактором 2, ~20 значений `le`.
  Значения «некруглые» (`0.000131072`, `0.001048576`, …) — зато точные,
  без ребакетинга и без второй гистограммы; Grafana это не смущает,
  задокументировать;
- **native/exponential** экспорт (этап 5, OTLP и Prometheus protobuf)
  берёт сетку как есть — представление совместимо по построению.

Гистограммы: `latkit_query_duration_seconds`,
`latkit_txn_duration_seconds`, опционально
`latkit_query_first_row_seconds` (время до первой строки, PLAN.md §1
«дополнительно») — за флагом `--first-row-hist`, по умолчанию выключено:
удваивает число рядов с лейблом `query`.

### Р25. Длительность запроса: `ts_ready`, при pipelining — `ts_complete`

Развязка долга Р16 (наблюдение несёт все четыре таймстемпа, выбирает
этап 4):

- **не-pipelined** unit (`LK_QO_PIPELINED` не стоит):
  `dur = ts_ready − ts_start` — модель PLAN.md §1 («от `Q` до `Z`»,
  «от Bind до ReadyForQuery после Sync»): сервер закончил и готов к
  следующему запросу;
- **pipelined**: батч делит один `Z` — привязка `ts_ready` к каждому unit'у
  приписала бы последним запросам батча ожидание всего батча. Честная
  пер-запросная длительность — `dur = ts_complete − ts_start`.
  Систематическое смещение между режимами (у `ts_ready` внутри — ещё и
  отправка `Z`) ≪ ширины бакета (±9%), но задокументировать;
- `ts_ready == 0` (ABORTED/CANCEL — `Z` ещё не видели) — в гистограмму
  **не** попадает: unit учитывается только счётчиком;
- `latkit_query_first_row_seconds` = `ts_first_row − ts_start`, только
  если `ts_first_row != 0`.

Точность по-прежнему ограничена моделью Р13 (таймстемпы — границы
syscall'ов): это «сеть-до-сети» время сервера, не `EXPLAIN ANALYZE`;
формальная сверка с `pg_stat_statements` — этап 8.

### Р26. Агрегация — в потоке событий, без локов; дамп — тот же поток

Обработчик `on_query` (агрегатор) вызывается синхронно из
`ring_buffer__poll` — в том же единственном потоке, что и весь пайплайн
(решение этапа 2: «агрегация под одним потоком, профилировать позже»).
Бюджет: нормализация + инкремент — микросекунды на запрос; при целевых
50k qps это единицы процентов ядра — приемлемо, замер в 4.5, оптимизация
(шардирование, RCU) — этап 8, если упрёмся.

Читатель метрик (HTTP этапа 5) живёт в **том же epoll-цикле** (Р9),
поэтому реестру не нужны ни локи, ни снапшоты: сериализация — просто
обход таблицы между событиями ringbuf. Этап 4 закладывает это контрактом:
`lk_metrics_dump(m, FILE*)` пишет **валидный Prometheus text exposition
format** — этап 5 оборачивает его в HTTP, ничего не пересчитывая.
Потребители дампа в этом этапе: флаг `--dump-metrics[=path]` (по SIGUSR1 и
при выходе), replay-харнесс (ассерты по строкам дампа).

### Р27. Self-метрики: существующие счётчики становятся рядами

Всё уже считается (строка 10-секундной статистики) — задача этапа только
donести значения до реестра. Источники подключаются как **провайдеры**:
`lk_metrics_add_provider(m, fn, ctx)` — колбэк, который при дампе
переливает свою статистику в ряды; агрегатор не знает про libbpf, ядро —
про метрики:

| Ряд | Источник |
|---|---|
| `latkit_ringbuf_dropped_total` | per-CPU карта `stats` ядра (сумма по CPU) |
| `latkit_events_total{dir}` | она же |
| `latkit_resync_total` | `lk_reasm_stats.resyncs` (имя застолблено в reassembly.h) |
| `latkit_parse_errors_total` | счётчик `parse_errors` PG-парсера (Р18) |
| `latkit_unknown_msgs_total` | `unknown_msgs` (Р18) |
| `latkit_queries_dropped_total{reason}` | `units_dropped_{resync,close,overflow}` (Р19) — reason=resync/disconnect/overflow; закрывает слепое пятно «обрыв посреди запроса» |
| `latkit_conns_evicted_total{reason}` | `lk_conn_table_stats` (LRU/idle) |
| `latkit_metric_series` (gauge) | сам реестр — фактическое число рядов |
| `latkit_queries_other_total` | наблюдения, ушедшие в bucket `other` — метрика честности top-K |
| `process_cpu_seconds_total`, `process_resident_memory_bytes`, `process_start_time_seconds` | `getrusage(2)` + `/proc/self/statm` — стандартные имена Prometheus, PLAN.md «CPU/mem агента» |

Строка 10-секундной статистики остаётся (её читают глазами), но
становится тонкой — печатает из тех же провайдеров.

### Р28. Лейбл `query`: усечённый нормализованный текст, полный — не здесь

- в реестре хранится **нормализованный** текст, усечённый до
  `LK_QUERY_LABEL_MAX` (256 символов, конфигурируемо `--query-label-len`);
  усечение по границе UTF-8-последовательности, суффикс не добавляем
  (fingerprint всё равно по полному потоку токенов, Р22);
- экранирование по правилам text format (`\\`, `\"`, `\n`) — при
  сериализации, не в хранении;
- **сырой** SQL в реестр не попадает вообще: полный текст — только в
  OTel-спаны/exemplars этапа 5, которые возьмут его из `lk_query_obs`
  в момент `on_query` (текст живёт только на время колбэка — контракт Р16);
  маскирование литералов, обещанное в README (PLAN.md §7), выполняется
  нормализацией по построению;
- флаги наблюдения, влияющие на текст: `NO_TEXT` → `query="<name?>"`
  не изобретаем, честно `query="other"`, kind остаётся в
  `latkit_queries_total{kind}`; `TEXT_TRUNC`/trunc нормализатора →
  ряд как обычно (fingerprint по префиксу), счётчик
  `latkit_queries_truncated_total`.

## Номенклатура метрик (фиксируется этим этапом)

```
# запросы
latkit_queries_total{db,user,kind,code}            counter  kind=simple|extended|function|copy_in|copy_out|cancel
                                                            code=ok|error|aborted|canceled
latkit_query_duration_seconds{query,db,user,code}  histogram  code=ok|error (Р24/Р25)
latkit_query_first_row_seconds{query,db,user}      histogram  за флагом (Р24)
latkit_query_rows_total{query,db,user}             counter  строки из CommandComplete
latkit_query_errors_total{sqlstate,db,user}        counter  без лейбла query (Р23)
latkit_queries_truncated_total                     counter
latkit_queries_other_total                         counter

# транзакции (материал on_txn, Р16)
latkit_txn_duration_seconds{db,user,status}        histogram  status=ok|aborted (T→I | E→I)

# соединения (материал conn_table)
latkit_connections_active                          gauge
latkit_connections_opened_total                    counter

# self (Р27)
latkit_ringbuf_dropped_total, latkit_events_total{dir},
latkit_resync_total, latkit_parse_errors_total, latkit_unknown_msgs_total,
latkit_queries_dropped_total{reason}, latkit_conns_evicted_total{reason},
latkit_metric_series, process_cpu_seconds_total,
process_resident_memory_bytes, process_start_time_seconds
```

Единицы — секунды/байты, суффиксы `_total` у счётчиков — по конвенциям
Prometheus; имена после M2 считаются публичным API (дашборды этапа 7).

## Структура модулей

Целевой срез после этапа:

```
src/
├── agent/                  # без изменений; events.c собирает цепочку
│   └── events.c            #   proto_pg → (tee: --queries логгер) → metrics
├── proto/                  # этап 3, не трогаем (API стабилен)
├── norm/
│   ├── norm_sql.c/.h       # лексер + fingerprint (Р22), zero-alloc,
│   │                       #   без I/O и без libbpf
│   └── ...                 # (после v1 — norm-модули других языков)
├── metrics/
│   ├── metrics.c/.h        # фасад: lk_metrics_new/cfg, query_sink (Р26),
│   │                       #   провайдеры (Р27), dump
│   ├── registry.c          # таблица рядов, словарь запросов, top-K LRU,
│   │                       #   doorkeeper, other (Р23)
│   ├── hist.c/.h           # экспоненциальная гистограмма (Р24)
│   └── selfstats.c         # getrusage//proc/self, process_* (Р27)
└── ...
third_party/
└── xxhash/                 # XXH3, один header (Р22)
tests/
├── unit/                   # + test_norm_sql.c, test_hist.c, test_registry.c
└── replay/                 # test_replay: ассерты по дампу метрик
```

`src/norm/` и `src/metrics/` — чистые (без libbpf, без I/O кроме
`dump(FILE*)`), линкуются в агент и в тесты, как `src/proto/`.
Зависимости: `metrics` → `norm`, `proto/proto.h` (тип `lk_query_obs`);
`norm` — ни от чего, кроме libc и xxhash.

## API между модулями (эскиз)

```c
/* src/norm/norm_sql.h */
#define LK_NORM_TEXT_MAX 1024

struct lk_norm_out {
    char  text[LK_NORM_TEXT_MAX]; /* канонический текст, м.б. усечён */
    __u32 text_len;
    __u64 fp;                     /* XXH3-64 полного потока токенов */
    bool  trunc;                  /* вход или text усечён */
};

/* Всегда успешна: мусор на входе — тоже токены. Границы — по построению. */
void lk_norm_sql(const char *sql, size_t len, struct lk_norm_out *out);

/* fuzz-харнесс (задел этапа 8) */
int lk_norm_fuzz_one(const uint8_t *data, size_t n);
```

```c
/* src/metrics/metrics.h */
struct lk_metrics_cfg {
    __u32 top_queries;      /* K, default 500 (Р23) */
    __u32 query_label_len;  /* default 256 (Р28) */
    __u32 max_session_dims; /* default 32 (Р23) */
    bool  first_row_hist;   /* default false (Р24) */
};

struct lk_metrics *lk_metrics_new(const struct lk_metrics_cfg *cfg);
void lk_metrics_free(struct lk_metrics *m);

/* Реализация lk_query_sink (on_query/on_session/on_txn) — втыкается в
 * lk_proto_pg_new() как выход парсера. */
const struct lk_query_sink *lk_metrics_query_sink(struct lk_metrics *m);

/* Провайдеры self-метрик (Р27): вызываются при каждом dump. */
typedef void (*lk_metrics_provider_fn)(void *ctx, struct lk_metrics *m);
void lk_metrics_add_provider(struct lk_metrics *m,
                             lk_metrics_provider_fn fn, void *ctx);
/* ...и сеттеры для counter/gauge, которыми пользуются провайдеры:
 * lk_metrics_set_counter(m, name, labels, v), _set_gauge(...) —
 * идемпотентная запись абсолютного значения. */

/* Prometheus text exposition format, валидный для этапа 5 (Р26). */
int lk_metrics_dump(struct lk_metrics *m, FILE *f);
```

Внутри `registry.c`: ряд = `{key: {fp | other, dim_id, code}, hist | u64}`,
открытая адресация, ключи интернированы; словарь запросов — отдельная
таблица `{fp → text, lru}` (Р23), ряды ссылаются на неё по fp.

## Риски этапа

| Риск | Митигция |
|---|---|
| Лексер разойдётся с реальным SQL (dollar-quoting, вложенные комментарии, экзотика) → рассыпание одного запроса на много fingerprint'ов | правила — по документации PG lexer'а; табличные тесты на каждое правило; fuzz-корпус; расхождение деградирует в лишние ряды, top-K ограничивает ущерб |
| Чурн top-K: ad-hoc поток вымывает рабочие запросы, метрики скачут | doorkeeper (допуск со 2-го появления) + слив вытесненных в `other` (суммы монотонны); счётчик `queries_other_total` показывает масштаб; K конфигурируем |
| Counter reset при возврате вытесненного fp собьёт наивные запросы в Grafana | `rate()`/`increase()` переживают reset по дизайну Prometheus; поведение задокументировано; при жалобах — poor-man's pinning горячих fp (этап 8) |
| Взрыв произведения лейблов (query × db × user × code) незаметен до прод-нагрузки | лимиты на каждое измерение (Р23), `latkit_metric_series` в self-метриках, стресс-тест кардинальности в 4.5 с ассертом на потолок RSS |
| «Некруглые» границы классических бакетов удивят пользователей | задокументировано с обоснованием (точность без ребакетинга, совместимость с native histograms); при отторжении — таблица замены сетки в одном месте (`hist.h`) |
| Слияние по усечённому префиксу (бюджет захвата 8 КиБ) склеит разные длинные запросы | флаг trunc → `queries_truncated_total` показывает масштаб; лечится бюджетом захвата (конфиг), не кодом |
| Стоимость нормализации на горячем пути при всплесках qps | zero-alloc однопроходный лексер; замер в 4.5 с зафиксированным ориентиром; кэш fp по указателю/длине сырого текста — заготовленная оптимизация этапа 8 |
| Смешение моделей длительности (Р25: ready vs complete) исказит сравнение pipelined/непайплайн нагрузок | смещение ≪ ширины бакета; выбор зафиксирован и задокументирован до того, как стал дашбордом; сверка с pg_stat_statements — этап 8 |
