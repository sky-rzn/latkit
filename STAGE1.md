# Этап 1 — слой захвата (BPF)

Детализация этапа 1 из [PLAN.md](PLAN.md) (§4, веха M1). Оценка: **1–2 недели**.

**Цель:** превратить PoC из этапа 0 в стабильный слой захвата: события привязаны
к соединениям, фильтры конфигурируемы, потери считаются и сигнализируются
наверх, объём захвата ограничен бюджетом. На выходе — поток событий, на который
этап 2 (reassembly) может опираться без доработок ядра.

**Не входит в этап:** таблица соединений в userspace, reassembly, парсинг
протокола (этапы 2–3); TLS (этап 6); unix-сокеты (v1.1); ветки CO-RE для ядер
< 6.x (этап 8, каркас уже заложен в `iter_first_seg`).

**Долги этапа 0, закрываемые здесь** (помечены в коде и `docs/notes-iov.md`):

- `POC_CHUNK` 256 байт → чанки 4 КиБ (`latkit.h:7`);
- учёт потерь ringbuf (`latkit.h:9`);
- IPv6 (`latkit.h:16`);
- multi-segment `ITER_IOVEC` (notes-iov, «Known limitations»);
- дубли на loopback (SEND+RECV одного payload) — решается фильтром по
  локальному порту (задача 1.3).

## Ключевые проектные решения

Решения зафиксированы здесь, чтобы не пересматривать их в середине этапа.

### Р1. conn_id — socket cookie

Из двух вариантов PLAN.md (5-tuple+netns vs указатель на `sock` + generation)
берём третий, строго лучший: **`bpf_get_socket_cookie(sk)`** — u64,
уникальный на сокет за время жизни системы, монотонный, не переиспользуется
(в отличие от указателя на `sock` и от 5-tuple с TIME_WAIT-reuse). Доступен
tracing-программам с ядра 5.12 — внутри нашего минимума 5.15.

5-tuple при этом не исчезает: он нужен для лейблов и отладки, но передаётся
**один раз** в событии `CONN_OPEN`, а не в каждом data-событии. Data-события
несут только cookie — событие становится меньше, ringbuf свободнее.

### Р2. Реестр соединений в ядре: `BPF_MAP_TYPE_LRU_HASH`

Карта `conns`: ключ — cookie, значение — состояние соединения:

```c
struct lk_conn_state {
    struct lk_tuple tuple;  /* для synthetic CONN_OPEN и отладки */
    __u32 seq;              /* счётчик событий (см. Р5) */
    __u32 dropped;          /* события, потерянные на reserve-fail */
    __u8  flags;            /* LK_CS_OPEN_SENT и т.п. */
    __u8  capture_mode;     /* бюджет, см. Р6 */
};
```

Именно **LRU**-hash: если `CONN_CLOSE` пропущен (потерян, агент рестартовал,
соединение открыто до старта агента), запись рано или поздно вытеснится сама,
и утечки в ядре не будет. Размер — 65536 записей (запас ×10 от типичного
`max_connections`).

Соединения, открытые **до старта агента**, обрабатываются лениво: если на
data-пути cookie нет в `conns`, запись создаётся на месте и перед data-событием
эмитится синтетический `CONN_OPEN` (с флагом `LK_F_SYNTHETIC` — парсер этапа 3
знает, что startup-фазу этого соединения он не видел).

### Р3. Жизненный цикл — tracepoint `inet_sock_set_state`, не sockops

`tp_btf/inet_sock_set_state` даёт `{sk, oldstate, newstate}` на каждый переход
TCP-состояния, работает на весь хост и не требует привязки к cgroup (sockops
требует `BPF_CGROUP_SOCK_OPS` attach — лишняя сущность, k8s-специфику отложим).

- `newstate == TCP_ESTABLISHED` + порт подходит → вставка в `conns`, событие
  `CONN_OPEN` с полным tuple.
- `newstate == TCP_CLOSE` + cookie есть в `conns` → событие `CONN_CLOSE`,
  удаление из `conns`.

Внимание: tracepoint может срабатывать в softirq-контексте — `pid`/`comm` там
**мусор**, не читать и не фильтровать по ним (pid берём только на
send/recv-пути).

### Р4. Ringbuf: фиксированные size-классы вместо variable-size

`bpf_ringbuf_reserve()` требует константный размер на этапе верификации.
Резервировать всегда `hdr + 4096` — значит тратить 4 КиБ ringbuf на каждое
событие, включая 5-байтовые `ReadyForQuery`; это умножает вероятность потерь.

Решение: **два size-класса** записи data-события —
`LK_CHUNK_SMALL = 256` и `LK_CHUNK_FULL = 4096`. Выбор класса — по
фактическому объёму к захвату (известен до reserve). `CONN_OPEN`/`CONN_CLOSE` —
третий, маленький фиксированный тип записи. Дискриминация в userspace — по полю
`type` в общем заголовке `struct lk_ev_hdr`, которым начинается каждая запись.

(Альтернатива — `bpf_ringbuf_output()` из per-CPU scratch-буфера: даёт честный
variable size, но добавляет копирование. Держим в уме как fallback, если
2-классовая схема окажется расточительной на реальном профиле.)

### Р5. Обнаружение потерь: глобальный счётчик + per-conn seq

Два независимых механизма:

1. **Глобальная статистика** — `BPF_MAP_TYPE_PERCPU_ARRAY` `stats` со
   счётчиками: `events_emitted`, `reserve_failed`, `bytes_total`,
   `bytes_captured`, `iter_unsupported`, `recv_state_miss`. Инкремент
   `reserve_failed` — при каждом неудачном `bpf_ringbuf_reserve`. Userspace
   читает периодически; в этапе 4 это станет `latkit_ringbuf_dropped_total`,
   пока — строка в stderr раз в N секунд.

2. **Точная привязка потери к соединению** — `seq` в `lk_conn_state`,
   инкремент через `__sync_fetch_and_add` на каждое событие этого соединения.
   При reserve-fail инкрементируется `dropped` в состоянии соединения, и
   **следующее** успешное событие получает флаг `LK_F_GAP`. Userspace видит
   либо дыру в seq, либо `LK_F_GAP` → соединение «грязное», парсер этапа 3
   будет ресинхронизироваться. В этапе 1 обязанность агента — только
   зафиксировать и залогировать факт.

Размер ringbuf делаем конфигурируемым (`--ringbuf-bytes`, степень двойки,
дефолт 8 МиБ) через `bpf_map__set_max_entries()` до load.

### Р6. Бюджет захвата: механизм в этапе 1, политика — в этапе 3

Идея PLAN.md «`DataRow` не копировать целиком» требует знать границы сообщений,
т.е. парсер, — а он в userspace и появится в этапе 3. Разделяем:

- **Механизм (этот этап):**
  - лимит захвата на один вызов send/recv — `--capture-limit`, дефолт 8 КиБ
    (= 2 чанка; сервер PG флашит выходной буфер порциями ≤ 8 КиБ, так что
    дефолт захватывает backend-поток практически целиком);
  - поле `capture_mode` в `lk_conn_state`: `LK_CAP_FULL` (дефолт) /
    `LK_CAP_HEADERS` (жёсткий срез до 64 байт на вызов). Userspace может
    выставить режим записью в карту `conns`.
- **Политика (этап 3):** парсер, находясь внутри большого результата
  (поток `DataRow`), переключает соединение в `LK_CAP_HEADERS` и обратно.

Критично для честности данных: даже когда payload обрезан, событие несёт
`total_len` реального вызова — reassembler этапа 2 продвигает позицию потока
на `total_len` и знает точный размер «дыры».

### Р7. Фильтр: только серверная сторона (локальный порт)

Этап 0 матчил «любой из портов == 5432» и на loopback ловил каждый payload
дважды (SEND клиента + RECV сервера). Меняем предикат на
**`skc_num` (локальный порт) ∈ карта портов** — захватывается только серверный
сокет. Это даёт:

- дедупликацию на loopback бесплатно;
- стабильную семантику направлений: `RECV` = frontend→backend (запросы),
  `SEND` = backend→frontend (ответы) — ровно то, что нужно модели latency;
- согласованность с моделью развёртывания (агент стоит на хосте PG).

## Формат события (v1)

Заменяет PoC-`struct lk_event` в `src/bpf/latkit.h`:

```c
enum lk_ev_type { LK_EV_DATA = 0, LK_EV_CONN_OPEN = 1, LK_EV_CONN_CLOSE = 2 };
enum lk_dir     { LK_DIR_SEND = 0, LK_DIR_RECV = 1 };

#define LK_F_TRUNC     (1 << 0) /* cap_len < total_len: срезано бюджетом */
#define LK_F_GAP       (1 << 1) /* до этого события были потери по соединению */
#define LK_F_SYNTHETIC (1 << 2) /* CONN_OPEN создан лениво, startup не видели */

struct lk_tuple {
    __u8  family;              /* AF_INET / AF_INET6 */
    __u8  _pad[3];
    __u32 netns;               /* inode netns: skc_net.net->ns.inum */
    __u8  saddr[16], daddr[16];/* v4 — в первых 4 байтах */
    __u16 sport, dport;
};

struct lk_ev_hdr {
    __u64 conn_id;             /* socket cookie */
    __u64 ts_ns;               /* bpf_ktime_get_ns */
    __u32 seq;                 /* per-conn, монотонный; дыра == потеря */
    __u8  type;                /* enum lk_ev_type */
    __u8  dir;                 /* enum lk_dir, для LK_EV_DATA */
    __u16 flags;               /* LK_F_* */
};

struct lk_ev_conn {            /* CONN_OPEN / CONN_CLOSE */
    struct lk_ev_hdr hdr;
    struct lk_tuple  tuple;
    __u32 pid;                 /* 0, если контекст ненадёжен (tracepoint) */
    __u32 conn_dropped;        /* в CLOSE: итог потерь по соединению */
};

struct lk_ev_data {            /* два size-класса: payload 256 или 4096 */
    struct lk_ev_hdr hdr;
    __u32 total_len;           /* полный размер send/recv-вызова */
    __u32 off;                 /* смещение чанка внутри вызова */
    __u32 cap_len;             /* байт в payload */
    __u32 _pad;
    __u8  payload[];
};
```

Один вызов send/recv > 4 КиБ (в пределах capture-limit) порождает **несколько**
data-событий с общим `total_len`, возрастающими `off` и последовательными `seq`.

### 1.4 Чанки 4 КиБ, multi-chunk, multi-segment iovec (~2–3 дня)

Самая верифицируемо-опасная задача этапа — закладывать день на борьбу с
verifier.

- [ ] `LK_CHUNK_FULL = 4096`, `LK_CHUNK_SMALL = 256`; выбор класса по объёму
      до reserve.
- [ ] Multi-chunk: ограниченный цикл до `capture_limit / LK_CHUNK_FULL`
      итераций (дефолт — 2), каждый чанк — отдельный reserve/submit со своим
      `seq` и `off`. Каждый чанк пишет в начало собственного `payload[]` —
      это сохраняет фиксированное смещение назначения и не злит verifier
      (перенос из notes-iov: «verifier cost of a variable-offset destination
      write»).
- [ ] Multi-segment `ITER_IOVEC`: обобщить `iter_first_seg` →
      `iter_read(msg, off, dst, len)`: ограниченный проход по `__iov[0..7]`
      (`nr_segs` через CO-RE), суммарно до capture-limit. Для RECV — сохранять
      в `recv_state` до 8 пар `{base, len}` на fentry (расширить
      `struct lk_recv_state`).
- [ ] `LK_F_TRUNC`, если суммарно захвачено меньше `total_len`.
- [ ] Обновить `docs/notes-iov.md`: снять «Known limitations» по 256 байтам и
      первому сегменту.

**Готово, когда:** `select repeat('x', 100000)` даёт цепочки событий с
согласованными `off`/`total_len`/`LK_F_TRUNC`; тестовая программа с
`writev` из 3+ сегментов в порт PG захватывается конкатенированно (сравнить
байты с отправленными); verifier чист на текущем ядре, `dmesg` пуст.

## Риски этапа

| Риск | Митигция |
|---|---|
| Verifier отвергнет multi-chunk/multi-segment циклы | ограниченные циклы с константными границами; каждый чанк — отдельный reserve с записью в начало payload; день заложен в 1.4; fallback — `bpf_ringbuf_output` из per-CPU scratch |
| `bpf_get_socket_cookie` недоступен в каком-то из контекстов | проверить в 1.1 на обоих путях (fentry и tp_btf); fallback — указатель `sk` + generation-счётчик в `conns` |
| LRU-вытеснение активного соединения из `conns` | размер с запасом ×10; при промахе data-путь просто пересоздаёт запись (synthetic OPEN), данные не теряются |
| Два size-класса не попадут в реальный профиль трафика | счётчики `bytes_total`/`bytes_captured` в `stats` дают профиль бесплатно; пересмотреть пороги по данным |
| `inet_sock_set_state` не покрывает какой-то путь закрытия | сверить счётчики OPEN/CLOSE с pgbench-ожиданием в 1.7; LRU страхует от утечки в любом случае |
